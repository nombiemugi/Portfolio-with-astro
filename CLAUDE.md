## Project

Personal portfolio site (Andres Soler) built with Astro 5, deployed to Netlify. Static output — no backend; the contact form uses Netlify Forms (`data-netlify` attributes in `src/components/ContactContent.astro`). The site is bilingual: English at `/`, Spanish at `/es/`.

## Commands

- `npm run dev` — dev server with hot reload
- `npm run build` — production build to `dist/` (a `prebuild` hook runs `npm install` automatically)
- `npm run preview` — preview the built output

There are no test or lint scripts in this repo. Netlify builds with Node 22 (`netlify.toml`).

## Architecture

Astro static site with three extras configured in `astro.config.mjs`: the React integration (used only for the shadcn-style card components in `src/components/ui/*.tsx`), Tailwind CSS 4 via the `@tailwindcss/vite` plugin (no tailwind.config file), and `@astrojs/sitemap`. `site` is set to the Netlify URL — canonical/hreflang/OG URLs derive from it.

### i18n — every user-facing string goes through the dictionary

Astro's built-in i18n is configured with `defaultLocale: "en"`, locales `["en", "es"]`, English unprefixed. The system lives in `src/i18n/`:

- `ui.ts` — flat string dictionary keyed like `nav.home`, `hero.resume`, with `en` and `es` blocks (kept in key parity). The locale-specific CV PDF is also a dictionary entry (`hero.cvFile` → `/HvTmpEnglish.pdf` / `/HvTmpSpanish.pdf`).
- `utils.ts` — `getLangFromUrl(Astro.url)`, `useTranslations(lang)` (returns `t()`), `localizePath(path, lang)` (adds `/es` prefix), `getAlternatePath(url, targetLang)` (same page, other locale — used by `LanguageToggle.astro` in the header).

Components self-localize: they call `getLangFromUrl(Astro.url)` internally rather than taking a `lang` prop. Pages under `src/pages/es/` are thin duplicates of the root pages (same components; the URL determines the language). When adding a page, add both the root and `/es/` versions; slugs must match across locales because the language toggle is a pure `/es` prefix swap.

### Content collections

Schemas in `src/content/config.ts`. Both collections are split by locale subfolder, so entry slugs look like `en/fit-app` / `es/fit-app`:

- `src/content/projects/{en,es}/<slug>.md` — one file per locale per project, same filename. Required frontmatter: `title`, `description`, `date`, `img`, `url`, `author`, `technologies`, `tags`; optional `order` (listing sort, fallback 999 — keep `order` in sync between locales). Detail pages are generated by `src/pages/projects/[slug].astro` and `src/pages/es/projects/[slug].astro` (`getStaticPaths` filters by slug prefix and strips it for the route param) and rendered through `MarkdownPostLayout.astro`.
- `src/content/experience/{en,es}/*.md` — listing-only (home + about pages). Numeric `startDate`/`endDate` (missing `endDate` = current job, sorts first), `location` enum stays in English (`Remote`/`Onsite`/`Hybrid`) — only its display label is translated via the dictionary.

Pages/components fetch with `getCollection('projects', ({ slug }) => slug.startsWith(`${lang}/`))`.

### Layouts and shared page bodies

- `src/layouts/Layout.astro` — base layout (Header, Footer, SEO: canonical, hreflang alternates for both locales, Open Graph). Pages pass `pageTitle`/`pageDescription`; optional `pageImage`.
- `src/layouts/MarkdownPostLayout.astro` — project detail wrapper, receives `frontmatter`.
- `AboutContent.astro` / `ContactContent.astro` hold the full about/contact page bodies so the `/es/` pages don't duplicate markup. `ProjectCard.astro` is the single card used by both the home "latest projects" section and the projects grid.

### Theming

Light/dark mode works via a `data-theme="dark"` attribute on `<html>`, persisted in `localStorage`. An inline script in `Layout.astro` applies it before paint; `src/components/Switch.astro` is the toggle. Colors are handcrafted CSS variables defined in `src/styles/global.css` (`--color10`…`--color120`, `--accent`, etc.) with semantic aliases (`--card`, `--muted`, `--border`) in `src/styles/tokens.css`. Components use them through Tailwind arbitrary values, e.g. `text-[var(--accent)]` — prefer these tokens over hardcoded colors.

### Conventions

- TS path alias `@/*` → `src/*` (`tsconfig.json`), e.g. `import { cn } from "@/lib/utils"`.
- SVGs in `src/assets/svg/` are imported directly and used as components (never import SVGs from `/public` — files there are for URL references only).
- Astro components in `src/components/`; React (`.tsx`) only under `src/components/ui/`.
